## Task definition (locked)

A **task** is the smallest change that has a checkable acceptance criterion
(≈ one test going green). It is the unit of work for exactly one Ralph loop: a
fresh-context agent should be able to complete one task in one pass. Too coarse
and a loop can't finish; too fine and per-loop overhead dominates.

Operationally, "fits in one pass" means a single build loop completes the task
*without crossing the context-pressure threshold* (~80% of the window — see
`01-ralph-loop.md`). Crossing it is proof the task was too coarse, and the
harness splits it (a refinement, not a re-plan).

## Storage: one file per task (locked)

The plan is a set of files, one per task:

```
specs/                      # user-authored specs (input to planning)
  *.md
specs/tasks/                # generated by the plan loop
  0001-parse-stream-json.md
  0007-loop-engine.md
```

Each task file is **YAML frontmatter + markdown body** — parseable like JSON for
the harness, editable like markdown for humans and the agent:

```markdown
---
id: 0007
status: pending          # pending | active | done | blocked
deps: [0001, 0005]       # task ids that must be `done` first
acceptance: "go test ./engine passes; stall detection halts after N no-change loops"
---
## Loop engine

Drive fresh `claude -p` invocations, parse stream-json, checkpoint per loop.
References: specs/01-ralph-loop.md §Iteration anatomy.
```

### Frontmatter schema

- `id` — stable numeric id (matches filename prefix).
- `status` — `pending` | `active` | `done` | `blocked`.
- `reason` — required when `status: blocked`. A recognized reason taxonomy:
  - `context-overreach` — too big for one pass; resolved by a fresh split
    (cheaply resolvable; see `06-orchestration.md`).
  - `new-scope` — a missing requirement; resolved by the batched re-plan.
  - `dependency` — waiting on another task/external thing.
  - `error` — repeated failure; escalate (see `attempts`).
- `deps` — list of task ids that gate this one (drives ordering; the harness
  never selects a task whose deps aren't all `done`).
- `acceptance` — the checkable criterion; feeds the agent's target and informs
  the test gate.

(OPEN: optional fields — `files` touched hint, `notes`, `attempts` counter.)

## Why this format

- **Cheap:** the harness parses frontmatter to pick the next unblocked task and
  injects *only that file* (plus named spec excerpts) into the loop — small,
  focused context.
- **Quality:** `acceptance` gives each loop a precise target; `deps` prevent
  building B before A.
- **Robust mutation:** the agent's only structured edit is the `status:` field —
  trivial, hard to corrupt — vs. hand-editing canonical JSON.

## Mutation ownership (locked)

Decision: **the agent writes its task's `status`** (option A), with the harness
cross-checking — *not* a separate structured verdict channel.

- The **harness owns**: task selection, dependency ordering, stall/progress
  detection, and the **final done decision** (via the real test command — the
  agent's status is advisory for correctness).
- The **agent owns**: the work, and flipping its task's `status` — `active` while
  working, `done`/`blocked` (+`reason`) on exit.
- **Cross-check / fallback:** the harness never depends solely on the agent
  remembering. It infers outcome from signals it owns — `git diff` (did work
  happen) and the test command (does it pass) — and when it ends a loop itself
  (e.g. context-pressure kill), it **writes the status/`reason` directly**. So
  the outcome is recorded whether the agent flipped it or not.

Rationale: a single human-readable source of truth (the task file), no parallel
verdict channel to reconcile. The agent's best-effort flip is fully covered by
the inference fallback the harness needs regardless (the context-kill case
produces no agent signal either way).

## Derived checklist (locked)

`IMPLEMENTATION_PLAN.md` is **generated** from the task files — a human-readable
nested `- [ ]` / `- [x]` checklist for eyeballing progress. It is never
hand-maintained; the task files are the source of truth.

## Plan lifecycle

- The **plan loop** reads `specs/*.md` (the non-task specs) and emits/updates
  `specs/tasks/*.md`: decomposing requirements into smallest-checkable tasks,
  assigning ids, wiring `deps`, writing `acceptance`.
- The plan is "complete" when every requirement in `specs/*.md` is covered by
  tasks and there are no gaps. (OPEN: how plan-completeness is judged — agent
  self-assessment vs. a coverage check vs. user approval.)
- The **build loop** consumes `specs/tasks/*.md` and executes them (see
  `01-ralph-loop.md`).
